# 项目详细进度和设计细则

## 1. 当前项目状态

当前仓库已经完成从 `vocab_os/` 子目录提升到根目录的统一入口整理，现阶段主要包含四条主线：

- FastAPI + SQLite + Async SQLAlchemy 的词库与学习后端
- 总览页 `frontend/index.html` 的单元浏览、词卡编辑、例句与笔记维护
- 沉浸式刷词页 `frontend/focus.html`
- 详情页 `frontend/focus_detail.html`

当前开发重点已经从“能跑起来”转到“刷词流程稳定、状态不丢、交互能闭环”。

## 2. 沉浸式刷词设计目标

### 2.1 目标

- 让用户在 `focus.html` 只做快速回忆与熟悉度判断
- 让用户在 `focus_detail.html` 查看释义、默认例句、笔记，再决定是否进入下一张
- 用 FSRS 管理正式评分与到期时间
- 保留单元模式和全局模式两种刷词范围
- 避免刷新、切换范围、回详情页时丢失当前会话

### 2.2 页面职责

`focus.html`

- 加载今日学习队列
- 展示正反卡片
- 只做翻牌与预选评分
- 展示会话内的新词/复习进度
- 展示当前单元或全局的总记忆进度

`focus_detail.html`

- 读取当前会话中的 `current` 卡片
- 展示释义、默认例句、默认例句发音
- 展示可编辑笔记
- 用 `下一个 / 记错了 / 上一个` 驱动会话流转

## 3. 当前刷词规则

### 3.1 队列规则

- `GET /api/db/study/today` 返回“今日新词 + 到期复习”的混合队列
- 新词和复习词的混排由后端 `_merge_study_queue(...)` 控制
- 当前会话只在评分为 `4` 时把卡片视为完成并从循环队列移除
- 评分为 `1 / 2 / 3` 时，卡片会带着更新后的 FSRS 状态回到队列尾部

### 3.2 详情页按钮规则

- `上一个`：查看刚刚学过的历史卡片，不推进当前队列
- `回到当前`：若正在查看历史卡片，则回到当前卡片详情
- `记错了`：强制按评分 `1` 提交
- `下一个`：按卡片页预选评分提交

### 3.3 今日记忆标记

- 评分 `1` 时会额外清掉该词的 `memorized_today`
- 这一步用于让“记错了”的词不再被视作今天已经稳住

## 4. 当前前端状态设计

### 4.1 会话状态来源

沉浸式刷词现在依赖两层浏览器存储：

- `localStorage`
  - `engwords_focus_selected_unit`
  - 用于记住从总览页带入的当前单元
- `sessionStorage`
  - `engwords_focus_sessions_v4`
  - 用于保存刷词会话本身

### 4.2 session bucket 设计

为避免全局刷词与单元刷词相互覆盖，会话现在按 bucket 保存：

- `global`
- `unit:Unit_10_Sub1`
- `unit:Unit_1_Sub3`
- 其他单元同理

每个 bucket 保存：

- `queue`
- `current`
- `scope`
- `currentUnit`
- `stats`
- `completed`
- `history`
- `progressOverview`
- `pendingRating`

### 4.3 为什么这样设计

旧实现的问题是：

- 单元刷词和全局刷词共用一个 key
- 切到全局后会把单元会话覆盖掉
- 刷新后会重新拉取队列，导致会话内进度显示被打回初值

改为 bucket 后：

- 单元会话和全局会话可以并存
- 切换范围时先持久化当前 bucket，再恢复目标 bucket
- 刷新页面时优先恢复 bucket，再决定是否重新拉取队列

## 5. 当前进度显示设计

### 5.1 今日进度

页面上有两个“会话内进度”：

- 今日新词
- 待复习

这两个值来源于：

- 后端 `study/today` 返回的 `new_remaining` / `review_remaining`
- 前端本会话内 `completed.newCount` / `completed.reviewCount`

显示公式：

- 新词显示：`newCount / (newRemaining + newCount)`
- 复习显示：`reviewCount / (reviewRemaining + reviewCount)`

这属于“当前会话进度”，不是全库累计进度。

### 5.2 总记忆进度

顶部横向信息条展示的是“总记忆进度”：

- 单元模式：该单元中 `memorized_past = true` 的词数 / 该单元总词数
- 全局模式：全库中 `memorized_past = true` 的词数 / 全库总词数

来源接口：

- 单元模式：`GET /api/db/words/{unit}`
- 全局模式：`GET /api/db/all_words`

容错规则：

- 若实时请求失败，不把进度直接清零
- 继续保留当前会话里已保存的 `progressOverview`

## 6. 当前 UI 细则

### 6.1 顶部信息区

顶部区域现在拆成三部分：

- 页面标题：`沉浸式聚焦刷词`
- 当前范围信息：显示中文单元名，或者隐藏
- 顶部总进度：横向信息条
- 操作按钮：`切到全局 / 切回本单元`、`返回总览`、`刷新今日队列`

### 6.2 卡片正反面

当前翻牌已经从 3D 旋转降级成稳定的正反面切换：

- 正面：单词、音标、发音按钮
- 背面：释义、默认例句、自定义例句、笔记只读内容

这样做是为了避免视觉翻转时内容被重绘覆盖，导致“翻牌了但看不到详情”。

### 6.3 详情页内容裁剪

当前详情页只保留：

- 释义
- 默认例句
- 默认例句发音
- 笔记编辑

当前不展示：

- Tatoeba 例句块
- AI 例句块
- 自定义例句块的复杂操作按钮

这是为了先把主刷词链路做稳定，再决定是否把高级编辑能力逐步加回。

## 7. 后端关键实现点

### 7.1 刷词接口

- `GET /api/db/study/today`
- `POST /api/db/study/review`

### 7.2 本轮修复过的问题

`POST /api/db/study/review` 曾经在返回下一张卡片时触发 SQLAlchemy `MissingGreenlet`：

- 原因：查询里预加载了 `Word.examples`，但没有预加载 `Word.notes`
- 结果：`_study_card_payload(...)` 访问 `word.notes` 时在异步上下文外触发惰性加载
- 修复：在 `backend/app.py` 的复习查询里补上 `selectinload(Word.notes)`

## 8. 已知风险和后续任务

### 8.1 仍需继续验证

- 今日新词进度在长链路操作后是否仍有边界条件
- 切换单元 -> 切全局 -> 再切回单元时，队列与进度是否完全符合预期
- 当队列空了以后刷新页面，恢复逻辑是否符合用户预期

### 8.2 建议后续继续做的事

- 给刷词会话加显式 session id，而不是完全依赖浏览器临时存储
- 给 `study/today` 增加更明确的“已完成数量”字段，减少前端自行推导
- 给刷词页加一次完整的浏览器自动化回归
- 把单元中文映射抽到单独共享文件，避免 `app.js` 和 `focus.js` 双份维护
